# 특이사항 보고서 양식 — 타 센터 이식 지침서

> 목적: 삽교/내포에서 쓰는 **카드형 특이사항 보고서 양식**을 다른 센터에서도 **글자·색·간격·태그까지 동일하게** 출력하도록 만드는 지침.
> 핵심 원칙: **양식은 사람이 손으로 그리지 않는다.** 양식 전체가 `build_daily_all.py` 스크립트 하나에 정의돼 있으므로, 다른 센터는 **같은 스크립트를 그대로 쓰고 데이터(cfg JSON)만 센터별로 바꿔 넣는다.** 이게 유일하게 "동일 출력"을 보장하는 방법이다.

---

## 1. 왜 이렇게 하나 (설계 사상)

보고서를 매번 HTML로 직접 쓰면 카드 여백·색·태그 위치가 조금씩 달라진다. 그래서 **양식(HTML 생성 로직)과 내용(그날 데이터)을 분리**한다.

```
[센터별 그날 데이터: cfg.json]  →  [build_daily_all.py]  →  [동일한 양식의 보고서 HTML]
        (매일 바뀜)                   (모든 센터 공통·고정)          (센터만 다르고 모양은 같음)
```

다른 센터 이식 = **build_daily_all.py를 그 센터 폴더에 복사** + **그 센터 데이터로 cfg.json 생성** + **같은 명령 실행**. 끝.

---

## 2. 파일 3종 (센터에 넘길 것)

| 파일 | 역할 | 센터별 수정 |
|---|---|---|
| `build_daily_all.py` | 양식 생성 엔진(태그·형광펜·카드 레이아웃 전부) | **수정 금지 — 그대로 복사** |
| `cfg.json` | 그날 데이터(이름·혈압·특이사항 원문) | **매일 새로 생성** |
| (선택) `encrypt_report.js` | 진입코드 암호화·업로드 | 센터 진입코드만 변경 |

`build_daily_all.py`를 손대면 양식이 갈라진다. **엔진은 공통, 데이터만 갈아끼운다**가 이 지침의 전부다.

---

## 3. 데이터 계약 (cfg.json) — 이 형식만 지키면 됨

```json
{
  "date": "2026.07.20",
  "weekday": "월",
  "people": [
    ["강유월", "혈압 146-90 / 체온 36.5", "오전 프로그램 참여 양호. 점심 식사 다 드심."],
    ["김명순", "혈압 118-68 / 체온 36.5", "특이사항 없이 편안하게 지내심."]
  ],
  "absent": ["박선오", "신화순", "이동주"],
  "gyeolseok": "박선오·신화순·이동주 (개인사정 미이용)"
}
```

**필드 규칙**
- `people`: `[이름, "혈압·체온 문자열", "특이사항 원문"]` 배열. **PDF 순서 그대로, 급여제공 인원 전원.**
  - 혈압·체온 문자열 안에 `/ 체온 36.5`처럼 **체온 숫자**가 들어가야 발열 카드(붉은 테두리)가 자동으로 잡힌다.
  - 특이사항은 **케어포 입력 원문 그대로.** 임의로 "특이사항 없음"으로 줄이지 말 것(원칙).
- `absent`: 결석/미이용 이름 배열.
- `gyeolseok`: 결석자 한 줄 설명(보고서 하단 우측에 표기).

**중요:** 이름·수치·약물·증상은 **손대지 않는다.** 스크립트가 알아서 태그·형광펜을 붙인다.

---

## 4. 실행 명령 (모든 센터 동일)

```bash
python3 build_daily_all.py cfg.json > 20260720_특이사항_보고서.html
```

이 한 줄이 **삽교든 내포든 새 센터든 똑같은 모양**의 보고서를 만든다.

---

## 5. 양식이 자동으로 해주는 것 (스크립트 내장 규칙)

이식받는 센터가 **따로 신경 쓸 필요 없는** 부분. 참고용으로만.

**(a) 상단 '오늘의 태그' 모아보기** — 그날 나온 태그별로 해당 어르신 이름을 묶어 상단에 요약.

**(b) 카드별 태그** — 각 어르신 특이사항 원문에서 키워드를 자동 추출해 태그 부착.
- 붉은군(warn): 낙상·낙상위험·부종·근력저하·어지럼·설사·혈변·상처·배회·공격성·망상·거부·통증 등
- 파란군(care): 물리치료·투약변화·안약·병원진료·응급 등

**(c) 발열 카드 강조** — 체온 **≥ 37.0**이면 카드 왼쪽 테두리가 붉게, 수치가 굵게. `⚠️ 검증 필요` (발열 기준 37.0은 센터 임상 기준에 따라 조정 가능)

**(d) 형광펜** — 원문 안의 핵심 구절(무릎통증, 집에 간다, 드레싱 등)을 노란/붉은 배경으로 강조. 원문은 그대로 두고 표식만 얹음.

**(e) 오검출 가드** — "거부 없이", "통증 없음", "넘어지지 않" 같은 **부정 문맥은 태그·강조에서 제외.** "하시려고 열심히"가 '고열'로 잘못 잡히는 것도 차단.

---

## 6. 분류 기준 (cfg 만들 때 참고) `⚠️ 검증 필요`

아래는 **주의/일반 판단 기준**이며, 고시·임상 기준이라 센터·시점에 따라 달라질 수 있어 검증이 필요하다.

- **혈압**: 수축기 ≥ 140 또는 ≤ 100, 이완기 ≥ 90 → 주의
- **체온**: ≥ 37.0 → 발열 주의
- **행동/신체**: 목욕거부·배회·낙상/낙상위험·부종·상처/욕창·설사/혈변·인지저하·공격성·통증호소·처치(물리치료·드레싱·안약)

이 기준은 스크립트의 태그 로직과도 연결돼 있으니, 센터가 기준을 바꾸려면 `TAG_DEFS`(스크립트 상단)를 함께 조정한다.

---

## 7. 새 센터 세팅 체크리스트

1. `build_daily_all.py`를 새 센터 작업 폴더에 **그대로 복사**.
2. 그날 케어포 3-6 PDF에서 **이름·혈압/체온·특이사항 원문**을 뽑아 `cfg.json` 작성(3장 형식).
3. `python3 build_daily_all.py cfg.json > 보고서.html` 실행.
4. 출력 HTML을 원본 PDF와 **대조**(이름·수치·인원수 일치 확인).
5. (선택) 진입코드로 암호화·업로드.

---

## [설계 노트]
- 양식을 "문서"가 아니라 "스크립트"로 규정한 이유: 손으로 그리면 센터마다 미세하게 어긋나지만, 같은 엔진에 데이터만 갈아끼우면 **픽셀 단위로 동일**해진다. 이식의 핵심은 "엔진 공통·데이터 분리".
- 분류 수치(혈압·체온 기준)와 발열 37.0은 임상/고시 변동 가능 정보라 `⚠️ 검증 필요`로 표시했다. 센터가 자기 기준을 쓰면 cfg 분류와 `TAG_DEFS`를 함께 맞춰야 양식과 내용이 일치한다.
- 다음 단계로 원하시면: 새 센터용 **cfg 자동 생성 프롬프트**(PDF만 주면 cfg.json까지 뽑는)를 별도로 만들어 드릴 수 있음.
