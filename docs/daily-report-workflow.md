# 내포소망주간보호 일일 특이사항 보고서 정리 프롬프트

이 문서 하나만 다음 세션에 넘기면 동일한 방식으로 재현할 수 있도록,
이 저장소(`care-report-naepo`, 내포소망주간보호센터)에서 실제로 사용 중인
파이프라인 기준을 정리한다. 진입코드는 보안상 이 문서에 적지 않는다 —
배포 시 `scripts/deploy.py` 호출 인자로만 전달한다(필요하면 담당자에게 확인).

## 0. 입력

케어포(dn.carefor.co.kr) 3-6.급여제공 리포트에서 출력한 "장기요양급여
제공기록지" PDF 1개. 다음 두 형태 모두 나올 수 있고, 파서가 둘 다 지원한다.

- **하루치 서식**: 수급자 1명당 본문 1페이지(+특이사항이 길면 별지 1페이지),
  날짜 열이 1개.
- **주간치 서식**(출력기간을 여러 날로 잡은 경우): 수급자 1명당 본문
  1페이지에 그 주 날짜들이 열로 나란히, 별지도 날짜별로 나뉘어 붙는다.

## 1. 인원 분류 — 3갈래로만 나눈다

PDF의 "총시간" 필드 값을 기준으로:

- **급여제공** — 총시간이 실제 분(分) 값 → 카드에 포함.
- **결석** — 총시간이 "결석" / "미이용" (또는 공란) → `absent` 목록에
  이름, 사유는 신체활동지원 특이사항에 "개인사유"/"개인사정" 문구가 있으면
  `"개인사정으로 인한 결석"`, 없으면 원문 상태어 그대로.
- **일정없음** — 총시간이 "일정없음" → 그날 서비스 대상이 아니므로
  급여제공에도 결석에도 넣지 않고 **전체 인원수 계산에서도 제외**한다.
  (주간 서식에서 특정 요일 열 자체가 없는 사람도 같은 취급.)

전체 표기: `전체 N명(=급여제공+결석) · 급여제공 N명 · 결석 N명`.

> ⚠️ **알아둘 것**: 2026.07.10 보고서(곽호성, 총시간 "일정없음")는 이 규칙이
> 정리되기 전에 배포되어 결석으로 집계되어 있다. 위 규칙대로면 전체 인원에서
> 제외되어야 한다 — 재수정 원하면 말해줄 것.

## 2. PDF 텍스트 추출 원칙

- `page.extract_table()`로 표 구조 그대로 읽는다(`scripts/parse_pdf.py`).
  라벨 열과 날짜별 값 열이 셀 단위로 이미 분리되어 있어, 세로쓰기 라벨
  ("신체활동지원" 등 한 글자씩 줄바꿈된 열)이 본문에 섞여 들어오는 문제가
  구조적으로 생기지 않는다.
- 메인 페이지의 좁은 "특이사항" 셀은 pdfplumber가 고정폭으로 줄바꿈하면서
  단어 중간에도 개행을 넣는다 — 줄바꿈을 공백 없이 이어붙인 뒤(`""`.join),
  사람이 자연스러운 띄어쓰기로 교정한다(3번 규칙). 별지첨부 페이지는 열
  폭이 넓어 줄바꿈이 실제 단어 경계와 맞으므로 공백으로 이어붙이면 된다.
- (참고용 대안: 표 구조가 깨끗하지 않은 PDF라면 `extract_words()`로 단어
  좌표를 얻어 세로 라벨 열(작은 x1 값)을 제외하고 `(top, x0)` 순으로
  재조립하는 방법도 가능하나, 지금까지 받은 PDF는 전부 표 구조가 깨끗해
  `extract_table()`만으로 충분했다.)
- 고정 서식 문구(특이사항/작성자성명/■/□/향상프로그램명/차량번호/대변·소변
  횟수 등)는 애초에 표의 다른 행·열에 있으므로 별도 토큰 제거가 필요 없다.

## 3. 특이사항 원문 정리 규칙

- 원문은 케어포 입력 그대로 게시한다. **요약·판단으로 "특이사항 없음"이라
  쓰지 않는다** — 신체활동지원/인지관리/건강간호 3개 항목이 전부 정말
  공란일 때만 "특이사항 없음"으로 쓴다(전원 카드 유지, 임의 생략 금지).
- 손댈 수 있는 것: 원본이 붙어 나온 띄어쓰기·오타 교정, 그리고
  요양(신체활동지원)↔간호 기록이 같은 사건을 중복 서술할 때 자연스럽게
  한 번만 남기기. 사실·수치·약물명·사람 이름은 절대 삭제하지 않는다.
  서로 다른 정보면 전부 보존한다.
- 기능회복훈련(4번째 항목) 특이사항은 위 3개 항목에 포함하지 않는다 —
  해당 항목만 공란이어도 다른 3개가 비어 있으면 "특이사항 없음"으로 본다.
- 원본 자체가 짧게 잘려 있는 경우("발톱", "손톱머리" 등)는 지어내서
  완성하지 않고 있는 그대로(공백만 보정) 둔다.
- 혈압/체온이 "/"(공백)로 찍혀 있으면 그날 미측정 → **"측정안됨"**으로
  표기한다(원문에 값이 없다는 사실 자체는 그대로 드러냄).
- 평가 기준·고시·수가 등 시점에 따라 바뀔 수 있는 정보가 섞여 있으면
  ⚠️ 검증 필요로 표시한다.

## 4. 데이터 구조 (cfg.json)

```json
{
 "date": "2026.07.13",
 "weekday": "월",
 "people": [
   ["이름", "혈압-확장/체온 또는 측정안됨", "정리된 특이사항 원문"]
 ],
 "absent": ["결석자1", "결석자2"],
 "gyeolseok": "결석자1·결석자2(개인사정으로 인한 결석)"
}
```

`people` 순서는 PDF(=`parsed.json`)에 나온 순서를 그대로 따른다(임의
정렬 금지) — 현재 파서는 이름 가나다순으로 정렬해서 내보내는데, 이는
PDF 자체가 원래 가나다순으로 편철되어 있기 때문이다.

## 5. 카드형 HTML 규칙 (`scripts/build_daily_all.py`)

- 전원 카드형, 주의/일반 구분 없이 모두 노출(일일 보고서 한정 — 주간·월간은
  주의 인원 위주로 다르게 구성, `scripts/build_weekly_all.py`).
- 카드 상단에 이름 + 혈압/체온. 수축기 ≤100·≥140, 이완기 ≥90,
  체온 ≥37.0 중 하나라도 해당하면 수치를 붉게 강조.
- 본문 강조(`<mark>`): 낙상·상처·설사·인지저하·공격적 등 위험 키워드는
  danger(붉은 배경), 통증·어지럼 계열은 info(노란빛 배경)로 형광펜 처리.
  **발열은 텍스트 키워드로 잡지 않는다** — "~하시려고 열심히" 같은 문구가
  오검출되므로, 체온 수치(≥37.0)로만 판단한다(`scripts/rules.py`).
- 안약·인공눈물·정기 투약처럼 매일 반복되는 일상 처치는 그 자체만으로는
  주의 신호로 보지 않는다(가족 대상 보고서에서 매번 뜨면 의미가 없음) —
  통증·낙상 등 실제 이상 신호가 문장에 있을 때만 주의로 잡힌다.

## 6. 주간 집계용 원천 데이터 (`data/YYYYMMDD.json`)

같은 사람 정보를 주의(`juui`)/일반(`ilban`)으로 재분류해 별도 저장한다.
주의 판정은 5번 카드 강조보다 엄격한 기준(문서 상단 10대 기준: 혈압/체온
이상·목욕거부·낙상위험·부종·상처/욕창·설사·인지저하·공격적·통증호소·처치)을
사람이 직접 적용해 태그를 붙인다 — `scripts/build_weekly_data.py`가
`cfg.json` + 사람이 작성한 판정 파일(judgments.json)을 받아 이 파일을
만든다. **일일 작업 때마다 빠짐없이 남겨야** 주간·월간 집계가 정확해진다.
`note`는 cfg.json의 원문 전체가 아니라 1문장 축약 요약으로 쓴다(기존
`data/*.json` 관행).

## 7. 카카오톡 알림 포맷 (아직 미구현 — 참고용 스펙만 기록)

일일 보고서는 상세 대신 업로드 완료 여부 + 인원 요약 + 링크만:

```
[내포소망] YYYY.MM.DD(요일) 특이사항 보고서 업로드 완료
급여제공 N명 (주의 N · 일반 N · 결석 N)

📄 https://yonzi0510.github.io/care-report-naepo/ (진입코드는 별도 확인)
```

주간·월간 보고서는 반대로 ⚠️ 주의 필요 대상자만 카톡에 풀어서 보여주고,
전체 내용은 이메일로 보낸다는 방침 — 실제 발송 자동화는 아직 만들지 않았다.

## 8. 최종 검증 체크리스트

- [ ] PDF 인원수 = 급여제공 + 결석 (일정없음은 별도 제외 확인)
- [ ] 보고서 인원 목록과 PDF 인원 목록 이름 대조 — 누락/중복 없음
- [ ] 혈압·체온·핵심 증상이 원문과 일치
- [ ] 배포 후 GitHub raw 캐시만 보고 안심하지 말고, 로컬에서 `python3 -m
      http.server`로 index.html을 띄워 진입코드로 직접 열어 당일 key가
      실제로 반영됐는지 확인한다(raw.githubusercontent.com은 캐시가 남을
      수 있음). GitHub API(contents 엔드포인트)로 확인해도 된다.

## 구현 파일 매핑

| 단계 | 스크립트 |
|---|---|
| PDF → 사람·날짜별 구조화 | `scripts/parse_pdf.py` |
| 구조화 데이터 → cfg 원재료(인원분류·결합) | `scripts/build_cfg_raw.py` |
| (사람이) 띄어쓰기 교정·중복 정리 후 cfg.json 확정 | 수작업 |
| cfg.json → 카드형 HTML | `scripts/build_daily_all.py` |
| cfg.json + 판정 → data/YYYYMMDD.json | `scripts/build_weekly_data.py` |
| data/*.json 여러 날 → 주간 카드형 HTML | `scripts/build_weekly_all.py` |
| HTML → AES-GCM 암호화 + manifest 갱신 | `scripts/deploy.py` |
| 주의 판정/키워드 규칙 공용 모듈 | `scripts/rules.py` |
